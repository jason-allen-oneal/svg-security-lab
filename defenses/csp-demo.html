Content Security Policy (CSP) Protection

✅ CSP Active: Content Security Policy is enabled and blocking script execution

This page has a strict Content Security Policy that prevents SVG files from executing JavaScript. The CSP header is:

<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'none'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; object-src 'none';">

What CSP Blocks:

  • script-src 'none': Blocks all JavaScript execution
  • object-src 'none': Blocks object/embed elements
  • default-src 'self': Restricts resources to same origin

CSP Implementation Guide

1. Basic CSP Header

Content-Security-Policy: default-src 'self'; script-src 'none'

2. Strict CSP for SVG Protection

Content-Security-Policy: default-src 'self'; script-src 'none'; object-src 'none'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:;

3. Report-Only Mode (for testing)

Content-Security-Policy-Report-Only: default-src 'self'; script-src 'none'; report-uri /csp-report

4. Nonce-based CSP (if scripts needed)

Content-Security-Policy: default-src 'self'; script-src 'nonce-random123'; object-src 'none'

Browser Console Analysis

Open your browser's developer console to see CSP violation reports:

// CSP violations will appear as warnings like: // Refused to execute inline script because it violates the following // Content Security Policy directive: "script-src 'none'"
💡 Tip: CSP violations are logged to the console, helping you identify and fix security issues.

Best Practices

  • Start Strict: Begin with a restrictive CSP and relax as needed
  • Use Report-Only: Test CSP in report-only mode before enforcing
  • Monitor Violations: Set up CSP violation reporting
  • Regular Updates: Keep CSP policies updated with new threats
  • Test Thoroughly: Ensure legitimate functionality still works
